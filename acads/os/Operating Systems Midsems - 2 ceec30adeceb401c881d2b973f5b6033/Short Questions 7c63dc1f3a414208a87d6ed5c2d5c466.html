Short Questions

1. What is synchronisation and define deadlock?

Synchronization is a technique used in operating systems to ensure that multiple threads or processes running concurrently do not interfere with each other and access shared resources in a controlled and predictable manner. This is achieved by using synchronization primitives such as semaphores, locks, and monitors.

A deadlock is a situation in operating systems where two or more processes are blocked because each process is waiting for a resource that the other process has, and neither process can proceed. This creates a circular wait and results in all processes being blocked indefinitely. Deadlocks can occur in systems with multiple processors or in systems with a single processor. They can be prevented by using various synchronization techniques such as locking, timeouts, and ordering of resource requests.

  1. What is semaphore, types of semaphore?

    A semaphore is a synchronization primitive used in operating systems to control access to shared resources. It is a variable that is used to signal the status of a shared resource and provides a mechanism for coordinating access to the resource among multiple processes or threads.

    There are two types of semaphores:

    1. Binary Semaphore: A binary semaphore is used to control access to a single shared resource. It has only two states: "locked" or "unlocked". A binary semaphore can be used to implement mutual exclusion, where only one process or thread can access a shared resource at a time.
    1. Counting Semaphore: A counting semaphore is used to control access to multiple instances of a shared resource. It has an integer value that represents the number of available resources. A counting semaphore can be used to manage resources that have a limited availability, such as memory buffers or printers.

    Semaphores are used to solve synchronization problems and to prevent race conditions and deadlocks in systems with multiple processes

  1. What is a Critical section problem? what are the solutions for critical section problem?

    The critical section problem refers to the problem of coordinating access to a shared resource among multiple processes or threads in a computer system. The critical section is a section of code that accesses a shared resource and must be executed by only one process or thread at a time, to ensure that the shared resource remains in a consistent state.

    Solutions for the critical section problem include:

    1. Mutual Exclusion: One solution to the critical section problem is to enforce mutual exclusion, which ensures that only one process or thread can enter the critical section at a time. This can be achieved using synchronization primitives such as semaphores, locks, or monitors.
    1. Progress: Another solution is to ensure that if a process or thread is blocked waiting for a resource, it does not prevent other processes or threads from accessing that resource. This is called making progress, and can be ensured by using synchronization primitives such as semaphores or monitors.
    1. Bounded Waiting: A solution to the critical section problem must also ensure that waiting processes or threads are released in a predictable order, to prevent indefinite blocking. This is called bounded waiting, and can be ensured by using synchronization primitives such as semaphores or monitors.
    1. Starvation-free: The solution should not allow a process or thread to be blocked indefinitely, even if it is waiting for a resource that is not currently available. This is called starvation-free, and can be ensured by using synchronization primitives such as semaphores or monitors.

    In a resource allocation graph, each process or thread is represented by a node, and each resource is represented by an edge. The allocation of a resource to a process or thread is represented by a directed edge from the resource to the process or thread.

    For example, consider a system with two processes P1 and P2, and two resources R1 and R2. If process P1 has been allocated resource R1, and process P2 has been allocated resource R2, the resource allocation graph would look like this:

    
    R1   R2
     \   /
      P1 P2

    If process P1 requests resource R2 and process P2 requests resource R1, the resource allocation graph would change to:

     
    R1   R2
     \   /
      P2 P1
      |  |
      v  v

    In this case, the resource allocation graph forms a cycle, indicating a potential deadlock situation. To prevent deadlocks, the operating system must detect such cycles and resolve the deadlock by releasing one of the resources.

  1. What is Virtual Memory ? Explain difference between Logical & Physical Address space?

    Virtual memory is a technique used by operating systems to provide an illusion of having more memory than is physically available in the system. This is accomplished by temporarily transferring data from a computer's RAM to its hard disk.

    A logical address space is the address space seen by a process, and it is the address space within which a process executes. The logical address space is the virtual memory seen by the process, and it is independent of the physical memory.

    A physical address space is the actual memory space available in the computer, and it is the memory space that is addressed by the hardware. The physical address space is the real memory available to the operating system, and it is used to store the data and code of running processes.

    The difference between the logical address space and the physical address space is that the logical address space is a virtual memory space, while the physical address space is the actual memory space. The logical address space can be larger than the physical address space, because the operating system can temporarily transfer data from the RAM to the hard disk, providing an illusion of having more memory than is physically available.

  1. Explain different types of File Attributes?

    File attributes are metadata associated with a file that provides additional information about the file and controls the way it can be accessed, modified, or executed. The types of file attributes vary between operating systems, but some common file attributes include:

    1. Read-Only: The read-only attribute prevents a file from being modified, deleted, or renamed. This attribute is often used to protect important system files and configuration files from accidental modification.
    1. Hidden: The hidden attribute makes a file invisible in the file explorer, and it is not displayed unless the user has chosen to show hidden files.
    1. Archive: The archive attribute is used by backup software to track which files have been backed up and which files need to be backed up.
    1. System: The system attribute indicates that the file is used by the operating system or an essential system component, and it is not intended to be modified by the user.
    1. Executable: The executable attribute indicates that the file is a program that can be run, and it is usually associated with files that have the .exe or .com file extension.

    1. Encrypted: The encrypted attribute indicates that the file's contents have been encrypted for security, and it is used by encryption software to protect sensitive data.

    These are some of the most common file attributes, and the exact set of file attributes available may vary depending on the operating system.

  1. Explain open(),read(),write(),lseek(),close() System calls?
    1. open(): The open system call is used to open a file for reading, writing, or both. It takes the name of the file and the mode of access as its arguments, and returns a file descriptor, which is a non-negative integer used to refer to the opened file in subsequent system calls.
    1. read(): The read system call is used to read data from an open file. It takes the file descriptor, a pointer to a buffer where the data will be stored, and the number of bytes to read as its arguments. It returns the number of bytes actually read, or a negative value in case of an error.
    1. write(): The write system call is used to write data to an open file. It takes the file descriptor, a pointer to the buffer containing the data to be written, and the number of bytes to write as its arguments. It returns the number of bytes actually written, or a negative value in case of an error.
    1. lseek(): The lseek system call is used to change the position of the file pointer within a file. It takes the file descriptor, the offset to move the file pointer, and a reference point as its arguments. It returns the new position of the file pointer, or a negative value in case of an error.
    1. close(): The close system call is used to close an open file. It takes the file descriptor of the file to be closed as its argument, and it returns zero on success or a negative value in case of an error.

    These system calls form the basic interface for reading from and writing to files in most operating systems. They are used to implement higher-level file I/O functions, such as fopen, fread, fwrite, fseek, and fclose in the C standard library.

  1. What is a race condition?

    A race condition is a situation in computer programming where the outcome of a program is dependent on the timing or interleaving of multiple threads or processes. In a race condition, a program may produce incorrect results because two or more threads or processes access shared data concurrently, leading to unexpected interactions.

    For example, consider a program with two threads that increment a shared variable. If both threads access the variable at the same time, one of the updates may be lost, and the result will be incorrect. This is an example of a race condition.

    Race conditions are a common source of bugs in concurrent or parallel programs, and they can be difficult to detect and reproduce. To avoid race conditions, it is important to use proper synchronization techniques, such as locks, semaphores, or atomic operations, to ensure that only one thread or process can access shared data at a time.

  1. What are the Deadlock necessary conditions? Explain?

    A deadlock is a situation in computer programming where two or more processes are unable to continue executing because each is waiting for a resource that is held by the other. To occur, four necessary conditions must be met:

    1. Mutual Exclusion: The resources being requested by the processes must be non-sharable, meaning that only one process can use the resource at a time.
    1. Hold and Wait: At least one process must be holding a resource and waiting for another resource that is currently held by another process.
    1. No Preemption: Resources cannot be taken away from a process once they have been acquired. A process must release its resources voluntarily.
    1. Circular Wait: There must be a circular chain of processes waiting for resources, such that each process is waiting for a resource held by another process in the chain.

    These four conditions are known as the "deadlock necessary conditions" and are often used to diagnose and prevent deadlocks in concurrent or parallel programs. To avoid deadlocks, it is important to design algorithms and systems that limit the occurrence of these conditions. For example, by avoiding circular waits, or by allowing resources to be preempted in some cases.
